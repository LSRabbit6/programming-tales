# 第九话 · 驿站三信使

> 本话的轴要先泼一盆冷水：**跨小岛的"要么都做、要么都不做"，在这个宇宙里买不到完美版——你能选的只是哪种赔法。** 驿站长灵枢兔手下三位信使性格迥异：老大锁着等，老二先预留，老三先斩后奏再道歉。第五话其实已经偷偷展示过第四种（也是最流行的一种），话尾给它们排一张全家福。

## 开场：单岛时代的黄金年代

先怀念一下美好的旧时光。同一座小岛的钱庄里，"转账"是个被惯坏的孩子：

```sql
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 'A';
UPDATE account SET balance = balance + 100 WHERE id = 'B';
COMMIT;
```

要么都成，要么都不成（**原子性**，ACID 的 A）。凭什么？凭这两笔操作落在**同一座岛的同一本账**上——后仓老管家私下记了撤销日志（**undo log**），中途出任何岔子，按日志原路擦掉，像没发生过。**原子性的底气，是"所有人都归我一个人管"。**

现在甲灵体的户头在东岛钱庄，乙灵体的户头在西岛钱庄。两座岛互不隶属，中间隔着一片**会断航的雾海**（第五话的超时三义性阴魂不散）。东岛扣完灵贝，雾海断了，西岛没入账——一百枚灵贝**凭空消失**。谁来"原路擦掉"？没有谁，因为**没有一本共同的账**。

灵枢兔的驿站，就是为这片雾海开的。三位信使，是三种"人造共同账本"的尝试。

> 📜 **现实界的记载**：这就是跨行转账/跨服务下单的原型题。单机原子性由数据库的 undo log 白送；跨机器后，它变成必须自己设计的东西。

## 信使老大 2PC：抱着锁箱立正等号令

第一位信使走进驿站时，怀里抱着一只上了锁的箱子，站得笔直。它的门派叫**两阶段提交（2PC）**，思路来自婚礼：

- **第一阶段（prepare，"你愿意吗"）**：找一位**协调者**当主持人，问东岛"能扣吗？"问西岛"能加吗？"两家各自把资源**锁住**、写好日志、举手："我愿意。"注意——举了手就不许反悔，锁也不许放。
- **第二阶段（commit，"我现在宣布"）**：主持人收齐所有"我愿意"，广播"正式执行！"；有任何一家说不行，广播"全体回滚"。

逻辑严密，账单也严厉：

1. **全程锁资源**：从举手到宣布，甲的灵贝被锁着，谁也动不了——吞吐量被锁跪了。第二话的队列，在这里排到了最贵的资源上。
2. **主持人是单点**：最恐怖的剧本——所有人举完手，主持人在宣布前**倒下了**。东岛西岛锁着资源面面相觑：提交还是回滚？**不知道，只能等**（blocking）。婚礼现场集体僵住，新人举着手站到天亮。
3. 所以 2PC 的地盘在**数据库圈内部**（**XA** 协议、跨库中间件）——机器少、网络近、锁得起的场合。互联网跨服务场景基本告别它：太慢，且谁挂大家等。

## 信使老二 TCC：先把灵贝冻进冰晶

第二位信使的招牌动作，是掏出一枚冰晶——它不真扣钱，先把钱**冻**起来。门派叫 **Try-Confirm-Cancel（TCC）**，把每个业务动作改造成三段式：

- **Try**：不真扣，先**预留**——把甲的一百枚灵贝封进冰晶，标成"冻结"（你银行卡里见过的"冻结金额"就是它）
- **Confirm**：各方 Try 都成功了，把冻结变成真扣——这一步**必然成功**，因为资源早就留好了
- **Cancel**：任何一方 Try 失败，各自敲碎冰晶解冻，像没发生过

订渡舟＋订客栈的组合最能看出它的好：先把舟位和床位**都预留**，都留到了再一起确认——不会出现"舟订了、客栈没了"的尴尬。相比 2PC，它不长时间锁数据库（冻结是业务层的标记），吞吐好得多。

账单在开发侧，而且很贵：

1. **每个业务动作都要会三招**——开发量 ×3，且"预留"这个语义不是所有业务都天然有（扣库存好冻结，"发一道传讯符"怎么冻结？）
2. 两个行话级的坑：**空回滚**（Try 没到、Cancel 先到——取消一个不存在的预留，要能识别）和**悬挂**（Cancel 执行完，迟到的 Try 才到——冻结了一笔永远没人确认的灵贝）。防这两个坑，靠第五话的老朋友：**全局事务 ID + 状态记录 + 幂等**。

## 信使老三 Saga：先跑后折返，鞠躬递道歉信

第三位信使最洒脱：接了差事拔腿就跑，出了错才折返回来，挨家鞠躬递道歉信。门派叫 **Saga**：长流程拆成一串**本地小事务**，每步**直接提交**、不锁不留——订渡舟，直接买；订客栈，直接订。哪一步失败了，就把前面已提交的步骤**反向补偿**回去：客栈订失败 → 执行"退舟票"。

一句话人格：**做了再说，错了道歉。**

它是互联网长流程（下单→扣库存→扣款→发货）的主流选择，因为快、不锁、每步都是普通代码。但道歉模式的账单最有哲学味：

1. **中间态对外可见**：舟票已买、客栈还没订的那几息，全大陆都看得见你的半成品状态（2PC 锁着不让看，Saga 大方展示）——业务必须容忍这个
2. **补偿本身也会失败**：退舟票的窗口也可能断航——所以道歉信必须可重试、幂等，重试到死进**差错池**转人工。第五话的对账体系，就是 Saga 的最后一道安全网
3. **有些事没法道歉**：传讯符发出去了收不回，灵贝打进别人户头要靠追讨。设计 Saga 的第一件事，是把"不可补偿的步骤"挪到**流程最后**——先做能反悔的，最后做泼出去的水

## 全家福：四种付账方式排一排

把第五话那位（消息 + 最终一致 + 对账兜底——连"事中协调"都懒得做，靠事后收敛）也请进驿站合影：

| | 一致性 | 性能 | 开发成本 | 一句话人格 |
| --- | --- | --- | --- | --- |
| 2PC | 强（锁到宣布） | 差 | 低（中间件代劳） | 抱锁箱立正等号令 |
| TCC | 较强（资源预留） | 中 | **×3，还有空回滚/悬挂** | 先冻进冰晶后动手 |
| Saga | 最终一致 | 好 | 中（每步配道歉信） | 先斩后奏 |
| 消息+对账（第五话） | 最终一致 | 最好 | 中（靠基建摊薄） | 白天记账晚上对 |

**选型判据两个问题**：① 业务能不能容忍中间态被看见？不能容忍 → 往左走（2PC/TCC）；能 → 往右走。② 每一步的"道歉信"好不好写？好写 → Saga；写不出 → TCC 预留。真实世界的分布是：**2PC 留在数据库圈，互联网业务大面积住在右边两格**——因为第五话教过，右边的中间态有整套对账制度兜底，不是裸奔。

## 谢幕

单岛时代，原子性是老管家送的；跨岛时代，**原子性变成了一份要自己签的协议，条款是你选的赔法**。三位信使没有优劣，只有账单形状不同：2PC 用吞吐买确定，TCC 用开发量买速度，Saga 用"容忍中间态＋写好道歉信"买洒脱。而工程师的成熟，就是从问"哪个最好"变成问"**这个业务出错时，我赔得起哪种**"。

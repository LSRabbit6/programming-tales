# 第八话 · 藏书阁的倒目录

> 本话的轴，是一条能推广到半个架构世界的公理：**读得快，都是写的时候买的单。** 藏书阁不过是把这条公理执行得最极端的地方。话尾有一段写给修 RAG 的读者：倒排 vs 向量——面试的必考题。

## 开场：一炷香 vs 三个弹指

信息大陆的**藏书阁**收着百宝集市三百万册货品图谱——每件货一册，标题写着它是什么。小灵体走到阁前的问询牌上写下"红色 连衣裙"，三个弹指之后，两百万册相关图谱按"相关程度"排好队送到面前。

如果这活儿交给后仓老管家 MySQL，它会这么干：

```sql
SELECT * FROM products
WHERE title LIKE '%红色%' AND title LIKE '%连衣裙%'
```

第三话的老朋友 `type: ALL` 登场——三百万册从头到尾逐册看一遍标题。**`LIKE '%xx%'` 连索引都用不上**（B+ 树索引只认前缀，中间匹配没门），一炷香起步。

同一件事，差两千倍。秘密不在腿脚，在一次**视角翻转**——而且你在第四话已经见过这个翻转的亲戚了。

> 📜 **现实界的记载**：真实电商搜索的响应在百毫秒量级，数据库全表扫中缀匹配则以分钟计；"LIKE 中缀匹配用不上 B+ 树索引"是 MySQL 的标准行为。

## 英雄登场：司书兔的倒目录

藏书阁的司书是灵枢兔的又一重身份：**司书兔**。它干的事只有一件——把目录**倒过来印**。

正常的存法长这样（行话叫**正排**）：

```text
图谱 1001 → "夏季 红色 连衣裙 雪纺"
图谱 1002 → "红色 高跟鞋"
图谱 1003 → "蓝色 连衣裙"
```

按册找词——想知道"哪些货含红色"，只能一册册翻。司书兔把它**倒过来**（**倒排索引**）：

```text
"红色"   → [1001, 1002]
"连衣裙" → [1001, 1003]
"雪纺"   → [1001]
```

按词找册——就像一本书末尾的索引页："递归，见第 38、117、203 页"。查"红色 连衣裙"变成：取出"红色"的名单、取出"连衣裙"的名单、**两个有序名单求交集**——弹指之间，跟阁里存三百万册还是三亿册几乎无关，**只跟答案的数量有关**。

停一息，把这个翻转和第四话对齐：已读功能是"消息的名单 → 人的进度"，藏书阁是"册的词表 → 词的册单"。**同一招：换一个主语来存，查询就从 O(全量) 塌缩到 O(答案)。**这一招的通用名字，叫索引。

那代价呢（英雄照例收费）：这本倒目录必须**提前印好**——每进一册新图谱，司书兔要把它的标题拆开，把册号插进"红色""连衣裙""雪纺"每一个词条的名单里。**写入变贵变慢，换查询便宜。**这就是本话公理的第一次现身：读得快，是写的时候买的单。回头看你已经收集了一族了：数据库索引、缓存预热、第四话的写扩散——全是"把活儿从读时挪到写时"。

## 第一道坎：中文没有空格——裁词刀

英文拆词天然有空格，中文是连着的："红色连衣裙夏季新款"——司书兔的**裁词刀**该从哪下手？"红色/连衣裙/夏季/新款"，还是"红/色连/衣裙……"？

这活儿叫**分词**，由分词器干（词典 + 统计模型）。它是中文搜索的地基，也是一堆经典事故的产地：

- 切法歧义："南京市长江大桥"——南京市/长江大桥，还是南京/市长/江大桥？
- **新词问题**：热词"绝绝子"刚出现时词典里没有，被裁成"绝/绝/子"——小灵体搜"绝绝子"什么都搜不到。所以司书兔要持续喂新词进词典
- 问询牌上的词和入库时必须用**同一把裁词刀**，否则你切你的我切我的，名单永远对不上——一个隐蔽度极高的经典事故

## 第二道坎：找到两百万册，谁排第一？——相关性这门玄学

交集求完，两百万册都含"红色"和"连衣裙"。**排序才是搜索的灵魂**——小灵体只看前二十册。

谁更"相关"？经典答案是一对配合了半个世纪的指标：

- **TF（词频）**：这个词在这册图谱里出现得越多，越相关——标题里两次"连衣裙"的，大概率比顺嘴提了一次的更是连衣裙
- **IDF（逆文档频率）**：这个词在**全阁**越罕见，越值钱——"的""新款"满阁都是，命中了不说明什么；"雪纺"只有 1% 的图谱有，命中就是强信号。**词的价值 = 它的稀缺度**

两者相乘再加工程修正（对超长图谱打折、对词频收益递减）——这套成熟配方叫 **BM25**，Elasticsearch 的默认打分器。每册得一个分，按分排队。

但要诚实地标出玄学所在：**"相关"没有客观正确答案**。BM25 说甲册比乙册相关，小灵体却总取乙册——谁对？小灵体对。所以现代搜索的排序最终被**点击数据**接管：把"谁搜了什么、看到什么、取了什么"喂给模型学排序。相关性从公式变成了投票——这也是为什么搜索团队养的算法师比司书多。

## 第三道坎：目录印好了，货一直在变

倒目录查得快的前提是"提前印好"，可集市每息都在上新、改价、下架。改一册图谱 = 它标题里**每个词的名单都要动**——倒排索引最怕原地修改。

工程解法很有性格：**不改，只追加**。新图谱先写进司书兔手边的一本"小册子"（新 **segment**），查询时同时查大目录 + 小册子、结果合并；后台定期把小册子们**合并**进大目录（segment merge）。代价是一个你在真实产品里见过的现象：**新货上架后要过几息才搜得到**——小册子按刷新间隔（refresh interval，默认 1 秒左右）才对查询可见。这套折中有个名字：**近实时（NRT）**——藏书阁向你坦白："我不实时，我近实时。"

规模化那步已经不用教你了：一本目录装不下三亿册 → 按册切**分片**，每片一本目录，查询扇出到所有分片、各自出前二十、汇总再排——第二话的发号员和第六话的分片思想原样回归。**Elasticsearch ≈ Lucene（单机的倒排索引库）+ 这套分片/副本/汇总的分布式外壳**——如果你在项目里对接过 ES，拆开就这两层。

> 📜 **现实界的记载**：Lucene 的 segment 只追加 + 后台 merge、NRT refresh_interval 默认约 1 秒、BM25 自 ES 5.0 起为默认打分器——均为公开文档事实。

## 写给修 RAG 的读者：倒排 vs 向量——必考题

如今流行的 RAG（检索增强生成）里，**向量检索**是标配（pgvector、各类向量库）——它和倒排是检索世界的两大门派，面试必考辨析：

| | 倒排（BM25） | 向量（embedding） |
| --- | --- | --- |
| 找什么 | **词面匹配**——你说的词，文档里真有 | **语义相近**——意思像，词可以完全不同 |
| "数据库慢查询"能召回"DB 响应迟缓"吗 | 不能（一个词都不重合） | 能（向量空间里挨着） |
| 精确术语/编号/人名 | **强项**（错误码、SKU、单号） | 弱项（语义模糊掉了精确性） |
| 可解释性 | 强（命中了哪个词一目了然） | 弱（"距离近"没法跟人解释） |
| 索引本质 | 词 → 文档名单 | 高维空间里建"附近的邻居"目录（HNSW 这类） |

结论也是面试的标准答案：**成熟的 RAG 用混合检索（hybrid）**——BM25 抓词面精确命中，向量抓语义泛化，两路结果融合重排（RRF 之类）。谁单用一路，谁就在对应盲区里漏召回。如果你手上的项目目前只用了一路，别慌——"我知道盲区在哪、演进方向是 hybrid"这句话本身，就是面试加分答案：**边界诚实 + 知道下一步**。

## 谢幕

藏书阁没有魔法，只有三笔预付款：**写入时**裁词建倒目录（买查询速度）、**离线**养词典喂点击模型（买排序质量）、**后台**默默合并 segment（买近实时）。小灵体落笔那三个弹指的轻盈，是司书兔在你看不见的时间里付清的账。

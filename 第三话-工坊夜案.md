# 第三话 · 工坊夜案

> 本话的轴是一句排查界的祖训：**猜是破案的开始，证据才是破案的结束——法器的全部作用，就是把"感觉慢"变成"看见哪条队满了"。** 上一话的七条队，这话当嫌疑人名单用。

## 案发

周一清晨辰时二刻，工坊群岛的警讯螺炸了：

> 【P1】订单查询信符 RT 均值 3.2 秒（平日 80 毫秒），错误率 4%，已持续五分钟

柜面伙计同步来报："客人说订单页转圈圈。"今天值守的，是背着一整袋照影法器的**侦探兔**。它深吸一口气——从这一秒起，它说的每句话都要有证据。

## 第一步：先看卷轴，圈定案发范围

侦探兔展开柜台后的**发光卷轴仪表盘**（现实界叫 Grafana，数据来自 Prometheus 这类采集灵）。看三个数：RT、QPS、错误率。

发现：**只有订单查询这一路信符慢，其他信符正常；QPS 没有暴涨**（排除"被洪流打了"）；慢是从上周五换新符阵（发版）后**逐渐**恶化的，今早的人潮把它压垮。

这一步的价值是排除法：不是全岛慢（排除灵脉/发号灵/机房），不是洪峰来袭（排除抢购式打挂）。**嫌疑人从七条队缩小到"这路信符自己的链路"。**

## 第二步：top——先问灶台忙不忙

侦探兔跳上十七号工坊，取出第一件法器 `top`。这是最古老也最快的一刀，它回答一个问题：**工坊在拼命干活，还是在干等？**

照出来的是：**灶台火力才 15%**。

这是个重要证据：工坊根本没忙！信符慢 + 灶台闲，指向一个方向——**伙计们不是在干活，是在等什么东西**。（反过来，如果火力烧到 400%，破案方向就完全不同：谁在烧灶？死循环？序列化？还是大扫除？）

## 第三步：jstack——把所有伙计当场定格拍照

侦探兔举起第二件照影法器 **jstack**：对着工坊一照，**所有伙计被当场定格**——每名伙计此刻站在哪行符文上、什么姿势。间隔十秒拍两张（对比才能区分"路过"和"卡死"）。

两张影像高度一致，触目惊心：

```text
"http-nio-8080-exec-1" ... WAITING
   at ...HikariPool.getConnection(...)
"http-nio-8080-exec-2" ... WAITING
   at ...HikariPool.getConnection(...)
（×190 名几乎一样的）
```

**200 名 Tomcat 伙计，190 名卡在同一行：`HikariPool.getConnection`——在索道口排队等去后仓的缆车。** 上一话的画面原样上演：伙计 200 名，索道只有 10 根（SpringBoot 自带的 HikariCP **默认只架 10 根**，没人改过），190 名伙计在索道口打盹。

伙计姿势速记（jstack 影像里最常见的三种）：`RUNNABLE` 在干活；`WAITING/TIMED_WAITING` 在等通知（等锁、等索道、等下游回话）；`BLOCKED` 在等进某扇上了闩的门。**一堆伙计 WAITING 在同一个地方 = 那条队满了。**

> 📜 **现实界的记载**：Tomcat 默认 200 工作线程、SpringBoot 2.x 起默认连接池 HikariCP 且 maximum-pool-size 默认 10——"200 工人抢 10 根管子"是真实生产环境里反复上演的经典配置事故。

## 第三步半：新手剧本——加宽索道，然后把火送给下游

按住第一冲动看一眼："索道不够？那架到 50 根不就完了！"

隔壁工坊的年轻值守真这么干过：索道改 50，重启，RT 短暂回落——**三分钟后后仓老管家的灶台烧到 100%，全岛所有信符一起慢了。** 警讯螺从一声变成三十声。

这一步教训值一次晋升答辩：**队列满往往不是队列太窄，是下游消化不动才堆起来的。加宽队列 = 把更多洪水直接送进已经快淹死的下游**——上一话说的背压，亲手把它的堤扒了。正确的反射弧是：**看见队满，先去下游找"为什么消化慢"，而不是加宽队。**

（索道默认 10 根小不小？确实偏小。但它此刻是**症状**不是**病根**——先治病，再谈体质。）

## 第四步：去下游——老管家的口供簿

回滚索道到 10 根，侦探兔下到后仓找证据。老管家有本**慢账口供簿**（现实界叫 slow query log，记录所有翻账超过阈值的单子），一翻，一个惯犯反复出现：

```sql
# Query_time: 0.83s
SELECT * FROM orders WHERE user_phone = '138****' ORDER BY create_time DESC;
```

单笔 830 毫秒。上一话的算术立刻能用：一根索道一秒只能过 1.2 单，10 根索道 = 全池 **12 QPS**——早高峰几百 QPS 的查询压过来，队不满才怪。**信符 RT 3.2 秒的真相：翻账本身 0.8 秒 + 排队等索道 2.4 秒。**

## 第五步：EXPLAIN——问老管家"你打算怎么翻"

侦探兔取出第三件法器 `EXPLAIN`，逼老管家交代翻账计划。关键一栏 `type` 赫然写着：**ALL——整仓全翻**，`rows: 3,120,000`。

翻译成人话：`user_phone` 这个字段**没建索引**，每次查询老管家把 312 万页订单账**从头到尾翻一遍**。破案了：上周五上的新功能"按灵讯号查订单"，开发时演武库里只有 5000 页账，翻一遍毫秒级，没人察觉；正仓 312 万页，翻一遍 800 毫秒。**数据量是慢查询的显影液——bug 上线那天就在，是数据量把它显影出来的。**

```sql
ALTER TABLE orders ADD INDEX idx_user_phone (user_phone);
```

索引 = 给这本 312 万页的账册加了目录。加完再照一次 EXPLAIN：`type: ref, rows: 12`。翻账 0.8 秒 → **3 毫秒**。卷轴上的 RT 应声跌回 60 毫秒。**索道数压根没动——瓶颈解除后，原配置绰绰有余。**这就是"先根因后扩容"的完整弧线。

> 📜 **现实界的记载**："测试库数据少、上线后全表扫描慢查询拖垮连接池"是线上事故复盘里出镜率最高的剧本之一；EXPLAIN 的 type=ALL 与 rows 列就是判定全表扫描的标准证词。

## 尾声彩蛋：两周后的幽灵毛刺（工坊大扫除篇）

案子结了两周，卷轴又见怪事：同一路信符**每隔十几分钟毛刺一次**，RT 突然 1.5 秒，几秒后自愈。灶台不热、索道不堵、账目正常——上一话的七条队全查了，没有嫌疑人。

这时才轮到 JVM 三板斧出场：

1. **`jstat -gc <pid> 1000`**（每秒打一行大扫除记录）：发现 **Full GC 每 15 分钟一次，每次停 1.4 秒**——对上了！Full GC 是工坊的"全场大扫除"，扫除期间**所有伙计原地立定**（Stop-The-World）——毛刺就是全场立定的 1.4 秒。
2. 为什么杂物涨这么快？**`jmap -histo <pid> | head -20`**（杂物普查：哪种东西占地最多）：`byte[]` 和某个 `OrderExportDTO` 霸榜。
3. 具体谁在造这些杂物？请出**诊断刀灵 Arthas**（阿里开源的线上法器，不停工不改符文就能上手术台）：`trace` 跟踪每步耗时、`watch` 看进出参数——定位到账房的"导出订单册"功能，**一次把 3 万行结果整个搬进堂屋拼册子**。每导一次，一坨几百 MB 的大件直冲后院（老年代），攒几次就触发全场大扫除。

修法不是调大扫除的规程（那是给正常体质的工坊做优化），是**改掉暴食**：导出改成分页流式写（一次搬 1000 行、写完就丢）。Full GC 从 15 分钟一次变成一天一次，毛刺消失。

> JVM 排查的经验骨架就这一条：**先证明是不是大扫除（jstat），再看是什么在吃地方（jmap histo），最后揪出谁在造它们（Arthas/堆转储）。** 上来就背 G1/ZGC 调参口诀的，多半还没找到病人。

## 侦探法器箱（本话名词钩子表）

| 法器/名词 | 它回答什么问题 | 一句话钩子 |
| :--- | :--- | :--- |
| Grafana + Prometheus | 慢到什么范围？从何时开始？ | 先圈案发范围再下机器 |
| top | 机器在拼命还是在干等？ | CPU 闲 + 接口慢 = 在等什么 |
| jstack | 每个工人此刻卡在哪行代码？ | 拍两张，190 人 WAITING 同一处 = 队满处 |
| 线程状态 | RUNNABLE 干活 / WAITING 等通知 / BLOCKED 等锁 | 读 jstack 的识字课 |
| HikariCP 默认 10 | SpringBoot 自带连接池（框架送的，配置是你的） | 默认值是本话反派 |
| 背压回送 | 加宽队列 = 送洪水给下游 | 扩容前先问下游为什么消化慢 |
| 慢查询日志 | 哪条 SQL 是惯犯？ | MySQL 的口供记录 |
| EXPLAIN / type=ALL | 这条 SQL 打算全表翻书吗？ | rows 312 万 = 没目录 |
| 索引 | 给书加目录 | 0.8s → 3ms，本话正义使者 |
| jstat -gc | 毛刺是不是 GC 停顿？ | Full GC = 全场立定大扫除 |
| jmap -histo | 什么对象在吃内存？ | 对象普查表 |
| Arthas | 不重启，现场看任意方法的耗时/参数 | 线上诊断的瑞士军刀 |
| Stop-The-World | GC 时所有工人原地立定 | 幽灵毛刺的真身 |

**本话公理**（挂进全季暗线）：排查 = 沿着七条队找最满的那条；而**最满的队常常只是下游生病的症状**——先找病根，再谈扩容。

## 下回预告

第四话《万灵信板的红点》：五百灵的信坛上，你发一条讯息，右下角跳出"已读 341"——为这枚小红点，工坊要替**每一条讯息记住五百位灵体各自读没读**，存账量瞬间乘五百。现实界的两大信使给出了两个完全相反的答案，而它们都是对的。

# 第六话 · 货架三劫

![第六话 · 货架三劫](assets/06-hotsearch.png)

> 本话的轴：**缓存是给仓库修的近道，而近道只有一种死法——人群在同一瞬间全部涌回主路。** 所以缓存设计的真正考题从来不是"命中时多快"，而是"**不命中的那一刻会发生什么**"。三种死法，三个名字，全在这一话。

## 开场：先算一笔"近道账"

周二戌时，大陆顶流伶人雀娘在千音阁的邸报上官宣离缘。一个时辰之内，这张邸报被查看**三十亿次**——折算下来，每一息都有近百万只小灵体伸手来拿同一张纸。

后仓老管家 MySQL 当然接不住——第二话算过，复杂点的查询，一位老管家几千下每息就到头了。接住这场洪水的是**货架**：**仓库是老管家的地盘，货架是门口快手灵 Redis 和各层伙计摆的近道**。那张邸报被第一只灵体查过之后就摆上了货架，后面二十九亿次查询根本不进仓库。

货架有个统治一切的数字：**命中率**。命中率 99% 时，仓库只见到 1% 的流量。听起来很美，但把它反过来读，你会看到本话所有恐怖故事的根源：

> 命中率从 99% 掉到 90%——只掉了 9 个点——**仓库的流量翻了 10 倍**。

而仓库的门脸、通道、老管家的手速，都是按"有货架"的日子置办的，它**从没见过真实流量长什么样**。货架越好用，仓库越退化，近道一断，主路瞬间核爆。掌柜兔灵枢搬了把椅子坐到仓库门口——记住这个结构，三种死法全是它的变体。

> 📜 **现实界的记载**：缓存=Redis/本地内存等，数据库=MySQL 等；"命中率掉 9 个点、DB 流量翻 10 倍"是每个缓存事故复盘里都会重算一遍的算术。本话的邸报场景与数字为教学虚构，微博热搜面对的正是同一类"热点读洪峰"。

## 劫一：击穿——热货的货架空了 0.1 息

那张离缘邸报的货架位设了五分钟保质期（**TTL**，到点自动下架，防止货太陈旧）。

戌时过五分，TTL 到点，货架清空。就在**下一个 0.1 息**里，十万个请求同时发现货架是空的——按照"没货去仓库拿"的朴素逻辑，**十万只灵体同时冲进仓库，拿的还是同一张纸**。仓库瞬间被打穿，这就是**缓存击穿**：一个热点 key 过期的瞬间，等于对仓库发起一次定向爆破。

荒诞的地方在于：这十万次进仓里，**九万九千九百九十九次是浪费**——第一只灵体把货补上架，其他人等一下就好了。所以解法顺着这个荒诞长出来：

1. **互斥锁 / 单飞（singleflight）**：货架空了，只放**一只灵体**进仓补货，其他九万九千九百九十九只**原地等货架**（拿不到锁就短暂等待重试）。第二话的队列和第五话的"宁可慢不可猜"在这里合流：宁可等半息，不可万灵踩仓。
2. **逻辑过期**：热货**物理上永不下架**，只在货签上写一个"逻辑过期时辰"；伙计发现签过期了，先把旧货递给客人，同时**后台悄悄**派人进仓换新——客人拿到的最多是旧了五分钟的邸报，但货架永远不空。账单：你要接受"短暂的旧"（还记得吗，这叫最终一致的亲戚）。
3. 对预知的热点（大促的手炉、热搜位的邸报）**提前续期 / 预热**——开抢前就把货摆满。

## 劫二：穿透——查一件仓库里根本没有的货

黑市的邪修灵写了个符咒脚本，拿**随机伪造的、根本不存在的货牌编号**疯狂查询铺子。

货架逻辑的盲区暴露了：货架只摆"存在的东西"。不存在的编号，货架上**永远**没有 → 每一次查询都老老实实进仓库 → 老管家翻一圈，没有 → 下次同样的假牌来，**再翻一圈**。货架形同虚设，仓库裸奔——这叫**缓存穿透**：请求像箭一样直接穿过货架层。

两个解法，各有账单：

1. **缓存空值**：仓库说"没有"，就在货架上摆块木牌"此货不存在"（短 TTL）。下次同编号来，货架直接答复。账单：邪修用的是**随机**编号，每个只来一次——你的货架会被百万块"不存在"的木牌占满。
2. **布隆过滤器**：在货架前面雇一位特殊的门卫灵，法器是一册神奇的花名册（一张巨大的位图 + 几道哈希咒）。它只会说两句话：**"绝对没有"**（可信，直接打发走）和**"可能有"**（放行去查）。所有真实存在的货牌提前录入花名册，随机伪造的编号几乎全被"绝对没有"挡在门外。账单：① 有小概率**误判**（说"可能有"但其实没有——所以它只能挡箭，不能当真相）；② 花名册**只进不出**（划掉一个名字很难表达），货频繁增删的铺子不好伺候。

## 劫三：雪崩——所有货架同一息垮掉

伙计半夜批量上架十万件新货，上架符咒给每件货的保质期都写了同一个数：3600 息。

第二天同一息，**十万个 key 集体过期**。仓库在毫无预警的情况下遭遇全量人流——这次不是一个 key 的定向爆破，是**全线塌方**：缓存雪崩。另一种触发方式更干脆：快手灵 Redis 的分身们整体病倒，所有近道瞬间消失。

雪崩的解法是一套组合拳，每一记你都见过它的兄弟：

1. **过期时间加随机抖动**：TTL = 3600 + random(600)。把"同一息的集体死亡"摊成一炷香的稀疏小雨——和第二话"同时是幻觉"反着用：**人为制造先后，消灭同时**。
2. **多级货架**：Redis 前面再加一层**本地货架**（每座工坊自己屋里的小货架，现实界如 Caffeine）。Redis 倒了，本地货架还能顶一阵。
3. **熔断限流保仓库**：雪崩已经发生时，在仓库门口立起第一话那道漏斗栅栏——放进能处理的量，其余直接拒绝。**宁可一部分小灵体看到"稍后再试"，不能让仓库死掉拖垮全街**（柔性可用，第三次出场）。
4. **Redis 自身高可用**：第一话给快手灵配的值班医生和分身（哨兵自动扶正、集群分片），这里正式上岗——货架层自己先别倒。

## 加餐：热点 key——没过期也能死

最后一个坑最隐蔽：那张离缘邸报的货架位**没过期、一直在**，照样可能出事。

为什么？第二话的知识直接推：每息近百万次查询打向**同一个 key**，而一个 key 只住在 Redis 集群的**一个分片**上——单分片就是那个单线程工人，十万上下每息到头。集群再大也没用，**热点 key 把集群退化成了单机**。

解法两招，都是"摊"：

1. **本地货架兜第一层**：每座工坊自己屋里记一份这张邸报，哪怕只存一息——近百万请求摊到一千座工坊，每座每息只需要问 Redis 一次，Redis 眼里的流量从近百万变成一千。**保质期短到一息，威力大到千倍**，因为热点的本质是重复。
2. **key 打散**：把 `dibao:12345` 复制成 `dibao:12345:1` 到 `:10` 十个副本，散在十个分片上，读的时候随机挑一个——单点变十点。账单：换新邸报时要改十份（一致性的活儿变多了）。

## 谢幕：设计货架 = 设计失效

掌柜兔把三劫排在一起看，它们是同一句话的三种念法：

| 死法 | 近道怎么断的 | 一句话解法 |
| --- | --- | --- |
| 击穿 | 一个热 key 到点下架 | 只放一人补货，其他人等 |
| 穿透 | 查的东西近道上根本不可能有 | 门卫花名册，"绝对没有"直接打发 |
| 雪崩 | 全体 key 同时下架 / 货架整体垮掉 | 抖动摊开 + 多级货架 + 栅栏保仓库 |

所以老练的掌柜看货架方案，第一眼不看命中时的风光，问三个问题：**这个 key 过期的瞬间会发生什么？查不存在的货会发生什么？货架整体消失的那一刻，仓库会看到什么？**——设计缓存，本质是设计它失效的那一刻。

（还有一道著名的题本话故意没展开：换新货时，货架和仓库谁先改、货架上是撤下还是换上？——"缓存一致性"，坑深到值一话番外，此处按下不表。）

## 本话名词钩子表

| 名词 | 它是谁 | 一句话钩子 |
| --- | --- | --- |
| 命中率 | 货架解决掉的请求比例 | 99%→90% = 仓库流量 ×10 |
| TTL | 缓存的保质期 | 本话三分之二的事故源头 |
| 缓存击穿 | 热 key 过期瞬间万人踩仓 | 定向爆破 |
| 互斥锁 / singleflight | 只放一人补货 | 99999 次仓库查询是浪费 |
| 逻辑过期 | 物理不过期 + 后台异步刷新 | 用"短暂的旧"换"永远有货" |
| 缓存穿透 | 查不存在的数据，缓存形同虚设 | 箭直接穿过货架 |
| 空值缓存 | "此货不存在"的牌子 | 账单：货架被牌子占满 |
| 布隆过滤器 | 只会说"绝对没有 / 可能有"的门卫 | 有误判、难删除，只能挡箭 |
| 缓存雪崩 | 集体过期或缓存宕机 | 全线塌方 |
| 过期抖动 | TTL 加随机数 | 人为制造先后，消灭同时 |
| 多级缓存 / 本地缓存 | 工坊自己屋里的小货架 | Redis 前面再修一条近道 |
| 热点 key | 每秒近百万请求打一个 key | 把集群退化成单机 |
| key 打散 | 一个 key 复制成十份散在多分片 | 账单：更新要改十份 |

**全季暗线第五块拼图**：第一话 每个英雄都收费 → 第二话 每个环节都排队 → 第四话 账单可从需求侧付 → 第五话 错误活不过一天 → 第六话 **设计缓存 = 设计失效**（推而广之：设计任何依赖，都要先设计它消失的那一刻）。

## 下回预告

第七话《万灵出工》：一万只 Agent 灵同时出工干活——每一只都可能原地转圈死循环、可能把灵贝当柴烧、可能一凿子把 `rm -rf` 刻进工坊的地基。沙箱怎么关住它们、调度怎么喂饱它们、trace 怎么盯住它们——信息大陆迎来它最吵闹的一个清晨。

# 第一话 · 百宝集市的抢购之夜

> 本话的轴只有一句话：**卖出去的不能超过一百件，同时铺子不能塌。** 你会看到六位伙伴依次登场，每位都解决一个危机——每位也都收一笔费。

## 开场：一张海报引来的十万灵体

信息大陆的百宝集市上，灵枢兔的铺子要过周年庆。它挂出一张海报：镇店的「月华手炉」，平日五千九百枚灵贝，庆典夜只卖两千九，**限量一百只**，戌时开抢。

兔子掌柜心想：也就几百个客人吧。结果海报被风灵传遍了整片大陆——戌时未到，铺子门前的云阶上已经落满了**十万只小灵体**，个个攥着灵贝袋，眼睛盯着门板。

戌时的钟声敲响。十万只手同时拍向柜台。

我们从掌柜最天真的开局，看这间铺子怎么塌。

## 第一波塌法：所有人直接冲进后仓

铺子的账全记在后仓——一位叫 **MySQL 的老管家**守着一屋子账册，认真、可靠、**但翻账有手速上限**：复杂点的账目，一息之间撑死翻几千页（现实界管这个速度叫 QPS）。

十万个订单直接涌向后仓，两件事同时发生：

**第一件：老管家被踩塌了。** 几千页的手速对上十万的洪流，通道瞬间堵死，后面所有人排队、催促、放弃——更糟的是，老管家还管着全集市**别的铺子**的账。一场抢购，瘫了整条街。

**第二件：多卖了。** 老管家的规矩是四步：翻账看余量 → 还有货 → 记减一笔 → 开单。可十万人同时来，两只灵体**同一瞬间**都翻到"还剩 1 只"，都判定"有"，都往下走——一百只手炉，开出了一百零一张单。庆典夜多卖的那几十只，要么真赔货，要么挨个道歉退灵贝，铺子招牌当夜就得挂到大陆热搜上。

记住这两种塌法的名字：**打挂**（洪流超过承受力）与**超卖**（并发缝隙里多卖的货）。这一整话，就是掌柜兔和这两只鬼搏斗的故事。

> 📜 **现实界的记载**：后仓老管家=数据库（MySQL），复杂查询下一台通常只扛几千 QPS（简单查询能高不少，但"改库存"这种活快不起来）；"两个请求同时读到库存=1"正是真实电商秒杀事故里最经典的超卖成因。

## 伙伴一号登场：Redis——"别烦老管家，先问我"

掌柜兔请来的第一位伙伴，是住在门口高台上的**快手灵 Redis**——它把小账全记在自己脑子里（现实界叫内存），一息之间能答十万问，就是脑容量不大、只装得下小而热的账。

打法叫**库存预扣减**：开抢前，把"还剩 100"这个数交给 Redis。十万个请求全去高台上扣数，扣到 0 为止；**扣成功的那一百位贵客，才有资格进后仓找老管家开单**。老管家从"被十万人踩"变成"只接待一百位"。

可超卖的鬼跟着爬上了高台：Redis 若也分三步"看-判-减"，缝隙照旧。Redis 的解法干脆——**把三步焊成一道符**（一段 Lua 咒文整体施放，施放途中谁也不许插队，现实界叫原子性）。一步之内没有缝隙，鬼没处钻。

对付并发缝隙，工程界历来两大流派：**要么消灭中间态**（像这道符，把几步焊成一步，让中间态根本不存在），**要么持久化中间态**（每步都留痕以便恢复，现实界叫 checkpoint）。这话先记下，往后几话它会反复回来。

**Redis 的账单**：① 脑容量金贵，只能记"库存数"这种小热账；② 它自己也是**会病倒的伙伴**——它一倒抢购就瘫，于是还得给它配值班医生和分身（哨兵/集群），铺子的开销又添一笔；③ 它说"扣到了"和老管家真正落账之间，仍隔着一段路——这段路上的幽灵，第三幕再说。

> 📜 **现实界的记载**：Redis 是真实存在的内存数据库，单机十万 QPS 量级；"预扣减 + Lua 原子操作"就是各大电商秒杀的标准起手式。

## 伙伴二号登场：MQ——"洪水入库，慢慢放闸"

一百位贵客进了后仓，看似太平。可开一张单不是一步——记账、锁优惠、算脚力费、通知货栈……一套下来小半刻钟；门外那九万九千九百位没抢到的，还在疯狂拍门刷新。前头是十万的洪水，后头是每息几百单的手速——**上下游差着两个数量级**。

第二位伙伴登场：**渠工灵 MQ**。它在高台与后仓之间修了一条水渠和一座水库——扣数成功的单子不直接冲进后仓，先扔进水库，**立刻**回贵客一句"抢到了，单子马上好"；老管家按自己的手速，从水库里一单一单舀着办。洪峰进库，闸门匀速放水。

贵客看到的，是"排队中"三个字。

**MQ 的账单**：① 水库里的单子可能**重复送到**——老管家必须会认单（同一张单办两遍等于办一遍，现实界叫幂等），否则有人收两只手炉；② 贵客问"我的单到底成没成"，答案在水里飘着，还得另立一块看板答复他；③ "单子马上好"这五个字，掌柜兔和账房先生吵了三天。

> 📜 **现实界的记载**：MQ=消息队列（Kafka/RabbitMQ 等），"削峰填谷"是它的经典岗位；你在 12306 上见过的"排队中"，就是那座水库的水面。

## 第三波塌法：那九万九千九百只灵体的怒火

抢到的一百位安顿好了。真正的大头是**没抢到的九万九千九百只**——它们的每次拍门刷新，仍然是一张真实的信符。更凶的是，人群里混着**傀儡党**：一只灵体驱着一群木傀儡，一息拍五百下门，比手拍的良民快一千倍。

如果所有这些信符都要跑到高台上才被拒绝，快手灵也会被无意义的洪流淹死。一条原则浮出水面：**把信符消灭在离客人越近的地方，代价越低。**

## 伙伴三号登场：漏斗栅栏军团——层层劝退

这不是一位伙伴，是一支**纵深布防的军团**，从大陆边缘到后仓门口，一层层削：

1. **门板第一刀**：抢购按钮拍一次就灰三息；庆典没开始时按钮根本是死的。海报和铺面画像这类**不会变的东西**，早被复制到大陆各地的驿站小仓（现实界叫 CDN）——十万灵体看海报这件事，压根不碰铺子本尊，由遍布各城的驿站扛。
2. **发号灵第二刀：限流**。铺子门口立着**发号灵**，规矩写在牌子上："一息只放一万张信符进来，其余的客客气气回一句'太火爆了，稍后再来'。"听着粗暴，但这是**保命哲学**：与其十万张信符一起死，不如伺候好一万张、礼貌拒绝九万张。这个思想叫**柔性可用**——铺子降级，但不倒下。发号的具体手法有令牌桶、漏桶（名字都很形象：匀速发通行证）。
3. **风纪灵第三刀**：同一只灵体一息拍五百下门？拉进黑册。傀儡党在这层被清掉一大半。
4. **快手灵第四刀**：真正走到库存面前的，只剩百分之一的精锐——库存归零后，Redis 直接回"没了"，连 Lua 咒文都不用念。

四层漏斗过完，十万洪峰到后仓门口只剩涓涓细流。**抢购架构的本质不是"更强地处理信符"，而是"更早地拒绝信符"。**

**漏斗的账单**：发号灵的阈值定多少？定低了误伤真客人（明明有货却被拦），定高了栅栏形同虚设——这个数没有公式，要靠**压测**逼出来：自己人先扮成十万灵体打自己的铺子。于是掌柜兔又多了一套演武工具和演练规矩要养。

> 📜 **现实界的记载**：CDN、网关限流（令牌桶/漏桶）、风控、缓存拒绝，就是真实秒杀系统的标准四层漏斗；"每层挡掉大部分流量"是所有大厂秒杀方案的共同骨架。

## 第四波塌法：对不上的账——幽灵手炉与夜账房

庆典散场，账房先生来对数：快手灵说扣了 100，后仓的账册里只有 97 张单。3 张单去哪了？

——有 3 位贵客扣数成功后，单子在水渠里赶上工坊换灯（服务重启），漂没了。（还记得吗：Redis 扣数和老管家落账之间隔着一段路，这就是那段路的路难。）

这 3 只手炉现在处于**幽灵状态**：快手灵认为卖了，货栈不知道该发给谁。

## 伙伴四号登场：夜账房——最终一致性的守夜人

解法朴素得让人安心：**定时对账**。每隔几刻钟，一位打着灯笼的**夜账房**把快手灵的扣数流水和后仓的订单逐笔核对，发现"扣了但没单"的，要么补开单子、要么把库存**回补**放回池子里再卖。

这背后是分布式世界里最重要的一次**认怂**：跨越两位伙伴（Redis 和 MySQL）想做到"每一瞬间都完全一致"，代价大到不可接受（要用分布式事务把所有人锁住，手速回到解放前）。于是工程师改了承诺——**不保证每一刻一致，保证最终一致**：中间允许几刻钟的幽灵态，靠夜账房兜底收敛。现实界的电商、支付、库存，天下大半系统都活在这个承诺下。

很多工程团队管这个动作叫 **reconcile**：不信任过程，请一个独立的检查者定期核对两边、修复漂移——生成结果过校验、资产清单对实况，全是同一个动作换了身衣裳。

**夜账房的账单**：它是"事后"的——幽灵态存在的那几刻钟里，客人可能看到"抢到了"又变"没抢到"（库存回补时的体验毛刺）。以及，夜账房自己也是段代码，也会出错——谁来对"对账"的账？（答：监控 + 告警，见最后一位伙伴。）

> 📜 **现实界的记载**：定时对账 + 库存回补是电商秒杀的标准兜底；"最终一致性"是分布式系统教科书里对 CAP 取舍的经典落地。

## 谢幕前最后一位：发光卷轴——看得见战场的掌柜

上面所有防线都有一个共同前提：**掌柜兔得看得见战场**。每层放进来多少、拒绝多少、快手灵应答快慢、水库堆了多深、夜账房发现几笔差异——这些数字实时映在柜台后那面**发光卷轴仪表盘**上，值夜的伙计盯着。任何一层的数字异常，都要**先于客人投诉**被发现。

有经验的工程师把链路追踪做到节点级，不是为了卷轴好看，是为了戌时三刻那一瞬，能在三十息之内回答一句："堵在哪一层。"

## 本话名词钩子表（每词一句：它是谁 + 它收什么费）

| 名词 | 它是谁 | 它收什么费 |
| :--- | :--- | :--- |
| 超卖 | 并发读写缝隙里多卖出去的货 | 本话的头号反派 |
| 库存预扣减 | 把库存放 Redis 先扣、DB 只接待成功者 | 多养一个 Redis，且引出对账问题 |
| 原子操作 / Lua | 把"读-判-减"焊成不可插队的一步 | 逻辑必须塞进一小段脚本里，复杂了写不下 |
| 削峰填谷 | MQ 当水库，洪峰入库慢放 | 重复消费要幂等、状态要另建表、用户要接受"处理中" |
| 幂等 | 同一操作做两遍=做一遍 | 每个消费端都要专门设计，白给不了 |
| CDN | 静态页面分发到离用户最近的节点 | 只能扛静态内容，动态请求帮不上 |
| 限流（令牌桶/漏桶） | 门口保安，每秒只放 N 个 | 阈值靠压测逼出来，定错了要么误伤要么失守 |
| 柔性可用 / 降级 | 宁可服务好一部分，不全体陪葬 | 产品体验受损，要跟业务方谈判 |
| 缓存击穿/穿透/雪崩 | 缓存的三种死法（本话只露了个脸，第六话主场） | —— |
| 最终一致性 | 不保证时刻一致，保证最终收敛 | 中间存在幽灵态，靠对账兜底 |
| 对账 / reconcile | 独立检查者定期核对两边、修复漂移 | 事后性 + 对账自己也要被监控 |
| 压测 | 自己人先打自己，逼出真实容量数字 | 一套工具 + 演练制度要长期养 |

**全季暗线今天露了三次脸，你可能已经发现了**：每位伙伴都收费。Redis 收运维费，MQ 收幂等费，限流收误伤费，最终一致性收幽灵态费。**所谓架构设计，就是在预算内雇一支你养得起的伙伴团队**——面试官问"你为什么选 X"，他真正想听的是你知道 X 的账单。

## 下回预告

第二话《一枚信符的一生》：把时间放慢一亿倍，跟着一枚信符从小灵体的指尖走到后仓账册——你会发现它一路上**排了七次队**，而"十万人同时点击"里的"同时"，根本不存在。

# 第四话 · 万灵信板的红点

<!-- 插图位 -->

> 本话的轴：**规模是一台乘法器——一个"小功能"乘上亿级小灵体，就是一套大系统；而有些账单，最便宜的付法是改需求。** 你会看到同一个红点，两家老字号给出完全相反的答案，且都是对的。

## 开场：东家的一句话

信息大陆最大的传讯工坊「万灵信板」，例行周会。东家说："加个已读呗，就传讯符旁边显示'已读'，隔壁百工坊都有。小灵体们催很久了。"

信板管事灵枢兔手下的帮工灵小算心想：不就一格印记吗。每道传讯符上刻一个凹槽 `is_read`，对方拆符时把凹槽点亮。当天下午就交工了。

私传场景——一只灵体传给另一只灵体——这个方案**完全正确**。一道符、一个读者、一格印记，天衣无缝。小算收工。

两旬之后，东家回来了："结社的传讯阵也要。要显示'已读 341'，点开能看**谁读了谁没读**——各家掌柜特别要这个。"

小算拨开算盘，脸色慢慢变了。灵枢兔在旁边看着，没拦，它知道这一课得让算盘自己讲。

## 第一波账单：乘法的恐怖

传讯阵和私传差在哪？**一道符不再只有一个读者。**

一个五百灵的结社阵，你往阵里投一道符，工坊要记住的不是一格印记，而是**五百只灵体各自读没读**——一道符的已读状态 = 一张五百灵的名单。

小算把算盘拨到冒烟：一个活跃的五百灵阵，一天一万道符。每道符 × 五百个阵众 = **一天五百万条已读记录，一个阵**。整片大陆有一百万个阵。一天五万亿条记录的刻写量级——而传讯符本身才一百亿道。

**已读功能的存储账单，是传讯符本身的五百倍。** 红点比符贵五百倍——这就是乘法器的威力：小功能 × 大规模 = 大系统。第一话的公理在这话有了推论——**账单不看功能多大，看它被乘了多少次。**

## 第二波：两家老字号，两个相反的答案

灵枢兔没有立刻画图纸。它先翻了现实界的邸报——这道题，那边交过两份著名的答卷。

> 📜 **现实界的记载**（两家的产品行为是公开可见的，内部实现我们只讲通行做法）：
>
> **答卷一（钉钉路线）：付全款。** 钉钉的群消息就是有"已读 341 / 未读名单"，精确到人。它选择真金白银地把这笔存储和写入扛下来——因为它的客户是**老板**，"我发的通知谁没看"这个需求值钱，值到公司愿意为它建专门的存储系统。功能即卖点，账单即成本，商业上算得过来。
>
> **答卷二（微信路线）：不做。** 微信群聊至今没有已读功能。这不是技术做不到，是产品层面砍掉了——微信官方多次公开表示，不想给收消息的人增加心理压力，从一开始就坚定不做（腾讯公关总监张军在公开回应里把话说死了："以后也不会"）。于是那台乘法器**根本没有开机**。

灵枢兔把邸报拍在小算面前，让它把本话最重要的一句话焊住：

> **"不做"也是一个架构方案，而且常常是最优雅的那个。** 管事的第一反应不该是"这怎么实现"，而是"这值得实现吗、能不能换个便宜的需求达到八成效果"。需求这一侧也是设计空间——很多工匠灵一辈子只在实现侧内卷，忘了账单可以从需求侧砍。

## 第三波：中间解——把"名单"换成"指针"

假设东家拍板："要做，但没有付全款的预算。"有没有介于"付全款"和"不做"之间的方案？

有，而且它是本话的技术主菜。回头看那个五百倍的账单，它源自一个假设：**为每道符存一张读者名单**（符的视角）。换个视角呢？

**别记"每道符谁读了"，改记"每只灵体读到了第几道"。**

给阵里每道符编一个递增的号（seq：1, 2, 3, …）。每个阵众只存**一个数字**：他读到的最大编号，叫他的**水位线**。张三灵的水位线 = 8210，意思是 8210 号及之前的符他都读过了。

于是：

- 存储：五百灵的阵 = **五百个数字**，跟符的数量无关。五万亿的账单塌缩成一个零头
- "已读 341"怎么算：数一下有多少阵众的水位线 ≥ 这道符的 seq
- 你每读到新符，只是把自己的水位线往前推一格——刻写也从"每道符 × 每个读者"塌缩成"每灵每次一格"

**从 O(符数 × 灵数) 到 O(灵数)**——这就是视角切换的威力：把"对象的状态"改存成"观察者的进度"。

而且这一招灵枢兔见过。它想起渠工灵的水库。

> 📜 **现实界的记载**：这个思想有个家族名字叫**水位线 / offset（偏移量）**。**Kafka 记录每个消费者"消费到第几条"用的就是 offset**——它从不给每条消息记"谁消费过"的名单；**TCP 的 ack 序号**（"我收到第 N 字节了"）是同款；连你手机播客的"续播进度"都是它——存进度指针，不存"每一秒听没听过"的名单。

## 第四波：中间解的账单（英雄照例要收费）

水位线这么妙，为什么付全款的钉钉路线还存在？因为便宜有便宜的短处，灵枢兔挨条记在账板上：

1. **它只能表达"连续已读"。** 水位线 = "这条线之前的**全部**读过"。小灵体跳着读呢？拉到阵底瞟了最新一道符、中间三百道没看——水位线一推，那三百道全被算成"已读"。要支持"精确到每道符"的语义（掌柜要点开看谁真没读通知），水位线给不了，还得回名单方案。**语义精度是这个方案卖掉的东西。**
2. **"已读 341"这个数怎么来，又是一道读写取舍题**：读的时候现数五百个水位线（**读扩散**：写便宜、读贵）？还是每次有灵推水位线，就给相关符的计数石 +1（**写扩散**：读便宜、写贵）？判据一句话：**谁发生得频繁，就让谁便宜。** 进阵刷符（读）远比投符（写）频繁的场景，宁可写时多干活。这对名词以后你会在所有 feed 流设计里反复撞见——现实界的微博时间线、朋友圈，同一道题。
3. 已读上报是小灵体的信板反复发的，风灵送信意味着同一份"我读到 8210"可能到两次——好在水位线天然**幂等**（取 max，推两次和推一次一个样）。注意这个巧合的珍贵：第一话里渠工灵的水库逼着老管家专门学"认单"才防住重复，这里数据结构自己就长成幂等的。**选对数据结构，能白嫖掉一整类分布式难题。**

## 谢幕：这道题的完整答案长什么样

后来大陆上有年轻管事来讨教"传讯阵的已读怎么设计"（现实界的面试官也爱问同一题："设计群聊已读功能"），灵枢兔给的是一个分层的答案，而不是一个方案：

1. **先谈规模**：私传是一格印记，结阵是乘法——符数 × 阵众数，先把账单算出来吓自己一跳
2. **再谈需求侧**：这个精度值多少钱？做掌柜生意的（钉钉，老板要抓落实）付全款存名单；做散灵生意的（微信）直接砍需求——"不做"入选方案列表
3. **再给中间解**：水位线 / offset，O(灵数) 的存储，代价是只有"连续已读"语义
4. **最后补细账**：已读数的读扩散 vs 写扩散（谁频繁谁便宜）、上报幂等（max 天然免疫重复）

四层里只有一层在动凿子。**这就是"大局观"的样子：方案不是选出来的，是算出来的，而且需求侧永远在牌桌上。**

## 本话名词钩子表

| 名词 | 它是谁 | 一句话钩子 |
| --- | --- | --- |
| 乘法器 | 小功能 × 规模 = 大系统 | 红点比消息贵 500 倍 |
| 砍需求 | 需求侧的架构方案 | "不做"常是最优雅的设计 |
| 水位线 / offset / seq | 存"读到第几条"的进度指针 | Kafka offset、TCP ack、播客续播，一家人 |
| O(消息×人) → O(人) | 视角切换：对象状态 → 观察者进度 | 名单换指针，账单塌缩 |
| 连续已读语义 | 水位线卖掉的精度 | 跳读表达不了——便宜的代价 |
| 读扩散 vs 写扩散 | 现算 vs 预存的取舍 | 谁频繁谁便宜；feed 流设计的万能钥匙 |
| 天然幂等 | max 操作推几次都一样 | 选对数据结构 = 白嫖分布式难题 |

**全季暗线第三块拼图**：第一话 每个英雄都收费 → 第二话 每个环节都排队 → 第四话 **有些账单要从需求侧付**。

第五话《夜账房》：一只小灵体在煎饼摊拍符付了三十枚灵贝，灵贝从他的钱庄划到摊主手里，要跨**四家互不隶属的商行**——任何一环的风灵抖一下，就可能出现"他付了钱、摊主没收到"的量子态。金融系统怎么在"会丢信的网络"上砌出"一枚不差的钱"？第一话客串的对账，这次是主角——灵枢兔要挑灯当一夜账房先生。

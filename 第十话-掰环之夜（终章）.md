# 第十话 · 掰环之夜（终章）

> 终章的轴：**分布式的一切难题都从"分家"开始——东西怎么分给 N 间工坊，才能分得均、搬得少、挂得起。** 这道题看着最朴素，却是前面九话里分片、集群、扩容所有话题脚下的那块地基。话尾附全季公理总表，温习从那里进。

## 开场：一次好心办坏事的扩建

万灯庆典前夜（现实界管这一晚叫双十一），快手灵 Redis 的记忆装在三间分坊里（编号 0、1、2），账目按最直觉的规则分家：

```text
工坊编号 = hash(key) % 3
```

`user:1001` 算出来归 0 号坊，`item:889` 归 2 号坊——简单、均匀、无可挑剔。

杂役看压测数据有点紧，好心添了一间新坊。规则自动变成 `% 4`。

然后，**血案**：`hash("user:1001") = 10007`，除以 3 余 2，除以 4 余 3——**换家了**。掰着指头算一遍你会发现：模数从 3 变 4，**约四分之三的 key 全部换家**。而"换家"在缓存的世界里意味着什么？请求照新规则去新家找——新家是空的。**命中率瞬间从 99% 掉到 25%。**

第六话的账立刻兑现：命中率掉这么多 = 后仓流量放大几十倍 = **人造缓存雪崩**。老管家当场被踩塌，整条街瘫痪。加机器——杂役手册里最善意的动作——引发了年度事故。

病根一句话：**取模分家把"归属"和"工坊总数"焊死了**，N 一动，全体重新洗牌。我们要的是：N 变的时候，**只有必要的一小撮账目搬家**。

## 英雄登场：灵枢兔把直线掰成环

那一夜，灵枢兔以**掰环人**的身份出场。它做的第一件事是换个空间想象：别把工坊排成一排取模，把哈希值空间（0 到 2³²−1）**首尾相接掰成一个环**，像钟表盘。这门手艺叫**一致性哈希**：

- **工坊上环**：每间工坊按自己的哈希值（比如对坊名哈希）站到环上的某个刻度
- **key 找家**：key 也哈希到环上某刻度，然后**顺时针走，遇到的第一间工坊就是家**

就这么简单。现在看它怎么优雅地解决扩建：

**添一间工坊 D**，它站到环上 B 和 C 之间——**唯一受影响的，是原本"顺时针奔向 C"、现在半路先遇到 D 的那一小段弧上的 key**。其他所有 key 的顺时针第一站没变，**家照旧**。搬家量从 75% 塌缩到约 **1/N**。拆一间同理：只有死者名下的 key 顺延给下一站，其他人无感。

**归属只跟"环上的邻居关系"有关，跟工坊总数无关**——焊死的那根线被剪断了。

## 第二幕：环的两个先天缺陷

方案落地，新问题照例排队入场（全季看到这，你应该条件反射了）。

**缺陷一：工坊少时，分赃不均。** 三间工坊哈希到环上，位置是随机的——可能 A 和 B 挨得很近，C 独占大半个环。C 累死，A、B 闲死（**数据倾斜**）。

**缺陷二：一间塌了，负担全砸给一个邻居。** C 塌了，它名下所有 key 顺时针**全部**涌向下一站 D；D 本来就有自己的活，瞬间双倍负载——**也塌了**；D 的负担又全给 E……第六话的雪崩换了个马甲回来了，这次叫**连环倒**（cascading failure）。

两个缺陷同根：**环上的"点"太少、太随机。**

## 英雄二号：撒分身——每间工坊的一百五十个影子

掰环人的第二式妙在四两拨千斤：**每间工坊不站一个位置，撒出 150 个"分身"（虚拟节点）随机落满整个环**——环上看是 450 个点，实际背后还是三间工坊。

两个缺陷同时痊愈：

1. **均匀**：150 个分身随机散布，大数定律出手——每间工坊分到的弧段总长趋于相等，倾斜消失
2. **挂得起**：C 塌了，它的 150 个分身从环上消失——这 150 段弧的 key **顺延给的是散布在全环的各家分身**，负担被**所有幸存工坊均摊**，而不是砸死某一个邻居。连环倒的第一张多米诺被抽掉了

一句话收纳：**虚拟节点用"多撒点"把随机性驯成了均匀性。**

## 现实名单：这枚环住在谁家

> 📜 **现实界的记载**：
>
> - **Cassandra、DynamoDB**：教科书级的一致性哈希 + 虚拟节点
> - **Redis Cluster**：用的是它的表亲——**16384 个哈希槽**（预分片）：先把 key 空间切成固定 16384 个槽，槽再分给机器；扩容 = 把一部分槽**成建制地**搬给新机器。思想同源（归属与机器数解耦、搬家最小化），实现更"可控"——搬哪些槽由管理员指挥，而不是哈希函数说了算
> - **Memcached 客户端**：ketama 算法，一致性哈希的经典产地
> - **Nginx / Envoy 的 ring hash 负载均衡**、**分布式存储的数据放置**——凡是"东西要分给会变多变少的机器"的地方，都能看到这枚环或它的亲戚

## 终章谢幕：分家三问

全季最后一块拼图，把这话浓缩成三个问题——以后你见到任何分布式系统，先问它这三问：

1. **分得均吗？**（倾斜 = 有人累死有人闲死，第六话的热点 key 是它的极端形态）
2. **变化时搬得少吗？**（搬家 = 服务抖动，取模的 75% 是反面教材）
3. **挂一台，其他人接得住吗？**（负担要摊给所有人，不能砸死邻居）

分得均、搬得少、挂得起——**分家三问**。答好了，上面九话的高楼才有地方站。
